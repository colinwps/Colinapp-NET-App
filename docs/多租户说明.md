# 多租户开关说明

Colinapp 内置「按需启用」的多租户数据隔离，单租户场景零成本，多租户场景一键开启。

## 1. 设计

- 所有业务实体继承 `EntityBase`，自带 `TenantId`（`long?`）字段。
- `AppDbContext` 为每个 `EntityBase` 实体注册全局查询过滤器：
  ```
  e => !e.IsDeleted && (!_tenantFilterEnabled || e.TenantId == CurrentTenantId)
  ```
  - `_tenantFilterEnabled` 来自配置 `MultiTenancy:Enabled`。
  - `CurrentTenantId` 来自 `ICurrentUser.TenantId`。
- `AuditSaveChangesInterceptor` 在新增数据时自动写入 `TenantId ??= 当前租户`，无需业务代码关心。

因此「是否隔离」只由一个开关决定，业务代码、查询、控制器都无需改动。

## 2. 当前租户从哪来

`ICurrentUser.TenantId` 解析顺序（见 `CurrentUser`）：

1. JWT 的 `tenant_id` 声明（**生产权威来源**）。登录时若用户的 `TenantId` 非空，`JwtTokenService` 会把它写入令牌。
2. 请求头 `X-Tenant-Id`（**开发/测试便捷入口**）。未建立租户用户时，可用它快速验证隔离效果。

> 生产环境若不希望客户端能通过请求头伪造租户，应移除 `CurrentUser` 中的 `X-Tenant-Id` 回退分支，仅信任 JWT 声明（或改为从子域名/网关注入）。

## 3. 如何启用

1. 配置 `src/Colinapp.Api/appsettings.json`：
   ```json
   "MultiTenancy": { "Enabled": true }
   ```
2. 为用户分配租户：给 `sys_user.TenantId` 赋值（可扩展租户管理模块维护）。该用户登录后令牌即带 `tenant_id`，其所有读写自动限定在本租户。
3. 重启应用。开启后：
   - 查询：自动追加 `TenantId == 当前租户`，跨租户数据互不可见。
   - 写入：自动打上当前租户标记。

关闭（`Enabled:false`）则退回单租户，过滤器短路，`TenantId` 仍写入但不参与过滤。

## 4. 验证（开发期，用请求头模拟租户）

```
# 开关开启后
POST /api/notice  Header: X-Tenant-Id: 1   → 该公告 TenantId=1
POST /api/notice  Header: X-Tenant-Id: 2   → 该公告 TenantId=2
GET  /api/notice  Header: X-Tenant-Id: 1   → 仅返回租户1的公告
GET  /api/notice  Header: X-Tenant-Id: 2   → 仅返回租户2的公告
```

## 5. 后续可扩展

- 租户管理模块（租户 CRUD、套餐、到期）。
- 登录页选择/按子域名识别租户。
- 超级管理员「全租户」视图（查询时 `IgnoreQueryFilters()` 或特殊租户上下文）。
