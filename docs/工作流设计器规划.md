# 工作流可视化设计器 · 规划

> 目标：把现有「固定节点顺序审批」升级为「用户在画布上画流程图」的图驱动工作流。
> 流程以 JSON 图（节点 + 边）存储，后端引擎按图推进。本文档是实施蓝图，分四个阶段落地。

## 1. 现状与差距

已有（简易版）：`WorkflowDefinition.NodesJson` 存节点**数组**，实例按 `CurrentNodeIndex` 线性推进，
支持会签/或签、驳回、撤销、待办已办。

目标形态与差距：

| 能力 | 现状 | 目标 |
|---|---|---|
| 流程结构 | 线性数组 | 有向图（节点 + 边） |
| 节点类型 | 只有审批节点 | 开始 / 审批(或签) / 会签 / 条件网关 / 抄送 / 结束 |
| 条件分支 | 无 | 排他网关 + 边上的条件规则 |
| 表单 | 自由文本 FormData | 可定义字段的简单表单（条件判断的数据来源） |
| 编辑方式 | 表单式节点列表 | 画布拖拽连线（LogicFlow） |
| 推进方式 | 下标 +1 | 沿出边遍历图 |

## 2. 技术选型

**前端画布：LogicFlow**（滴滴开源，MIT）。
理由：专为流程图/审批流场景设计，自定义节点/边容易，数据模型天然是 `{nodes, edges}` JSON，
中文文档完善，Vue 3 可直接集成（`@logicflow/core` + `@logicflow/extension`）。

备选对比：
- AntV X6：更通用更强，但流程语义要自己搭，工作量大；
- bpmn-js：完整 BPMN 2.0 但存 XML、概念重，超出「简易引擎」定位；
- 钉钉式树形设计器：交互最傻瓜但不是自由画布，与「画出来」的目标不符。

**后端不引入工作流框架**（Elsa/Workflow Core 均过重且有版本/依赖风险），
继续自研：图存 JSON、引擎是纯函数式的「找当前节点 → 沿边推进」，代码量可控、可测试。

## 3. 流程 JSON Schema（核心设计）

流程图存于定义的 `GraphJson`（替代现 `NodesJson`），发起时快照进实例（沿用现有快照机制）。

```jsonc
{
  "version": 1,
  "nodes": [
    { "id": "start1", "type": "start", "name": "开始", "x": 80, "y": 200 },
    {
      "id": "n1", "type": "approval", "name": "主管审批", "x": 240, "y": 200,
      "props": {
        "approveMode": 1,          // 1=或签 2=会签（画布上「审批节点」「会签节点」是同一类型的两个预设）
        "approverType": 1,         // 1=指定用户 2=角色
        "approverIds": [1, 2],
        "emptyStrategy": 1         // 审批人解析为空时：1=自动跳过 2=转管理员（阶段四再扩）
      }
    },
    { "id": "g1", "type": "condition", "name": "金额判断", "x": 400, "y": 200 },
    { "id": "n2", "type": "approval", "name": "总经理审批", "x": 560, "y": 120, "props": { } },
    { "id": "cc1", "type": "cc", "name": "抄送人事", "x": 560, "y": 280, "props": { "userIds": [3] } },
    { "id": "end1", "type": "end", "name": "结束", "x": 720, "y": 200 }
  ],
  "edges": [
    { "id": "e1", "from": "start1", "to": "n1" },
    { "id": "e2", "from": "n1", "to": "g1" },
    {
      "id": "e3", "from": "g1", "to": "n2", "name": "大额",
      "priority": 1,
      "condition": {                       // 结构化规则，不做表达式 eval（安全、可用 UI 搭建）
        "logic": "and",
        "rules": [ { "field": "amount", "op": "gt", "value": 5000 } ]
      }
    },
    { "id": "e4", "from": "g1", "to": "cc1", "isDefault": true },  // 网关必须有一条默认边兜底
    { "id": "e5", "from": "n2", "to": "end1" },
    { "id": "e6", "from": "cc1", "to": "end1" }
  ]
}
```

要点：

- **条件挂在边上**（BPMN 风格），条件网关节点本身无属性；出边按 `priority` 求值，都不满足走 `isDefault` 边。
- **条件是结构化规则**（字段/操作符/值 + and/or 组合），后端安全求值，前端可做成规则搭建器；
  不做字符串表达式解析（有注入与维护成本）。操作符集合：`eq/ne/gt/ge/lt/le/in/contains`。
- **会签不是独立类型**：画布调色板给「审批节点」「会签节点」两个入口，落到 JSON 都是
  `type: "approval"`，仅 `approveMode` 预设不同。引擎只认一种审批节点，复杂度不扩散。
- **x/y 只服务于画布回显**，引擎忽略。LogicFlow 的原生导出格式与此结构做一层轻量转换（adapter）。

### 表单字段（条件分支的前置）

条件要有数据可判，`FormData` 必须从自由文本升级为结构化。定义上新增 `FormFieldsJson`：

```jsonc
[
  { "key": "amount", "label": "金额", "type": "number", "required": true },
  { "key": "reason", "label": "事由", "type": "textarea" },
  { "key": "leaveType", "label": "类型", "type": "select", "options": ["年假", "事假"] }
]
```

发起页按此渲染动态表单，提交值存 `FormData`（JSON 对象），求值时按 `field` 取值比较。
字段类型先支持：`text / textarea / number / date / select`。

## 4. 后端引擎改造

### 数据模型（改动小）

- `WorkflowDefinition`：`NodesJson` → `GraphJson`，新增 `FormFieldsJson`；
- `WorkflowInstance`：`CurrentNodeIndex(int)` → `CurrentNodeId(string)`，`NodesJson` → `GraphJson`（快照）；
- `WorkflowTask`：`NodeIndex` → `NodeId(string)`；其余（状态机、待办已办、撤销）全部沿用；
- 新增 `WorkflowCcRecord`（抄送记录：InstanceId / NodeId / UserId / ReadTime），支撑「抄送我的」列表。

### 引擎核心（Application/Workflow 内，纯逻辑可单测）

```
提交:   校验图 → 从 start 的出边找到首个「落脚节点」→ 生成任务
推进:   AdvanceFrom(instance, nodeId):
          沿 nodeId 出边取下一节点，循环处理直到停在审批节点或结束：
            approval → 解析审批人生成任务，停
            condition → 按 priority 求值出边，无命中走默认边
            cc       → 写抄送记录，继续沿出边走
            end      → 实例置为已通过
审批:   与现逻辑一致（或签推进/会签集齐推进/驳回终止），推进改调 AdvanceFrom
```

### 保存时的图校验（引擎健壮性的关键）

1. 恰好一个 `start`，至少一个 `end`；
2. 除 `end` 外每个节点至少一条出边；`start` 无入边、`end` 无出边；
3. 所有节点从 `start` 可达（DFS），无孤岛；
4. `condition` 节点 ≥2 条出边且必须有一条默认边；条件边引用的 `field` 必须在表单字段中存在；
5. `approval` 节点必须配审批人；
6. **禁止环**（拓扑检测）。简易版流程必须是 DAG，「驳回回退重审」用阶段四的驳回策略实现，不用图上的回环，避免引擎处理循环计数。

### 兼容迁移

现有线性定义/在途实例做一次性数据迁移：数组 `[A,B]` 机械生成
`start → A → B → end` 的图（坐标自动横排），`CurrentNodeIndex` 映射为对应 NodeId。
迁移代码写在 EF Migration 的 `Up()` 里，风险低（现在只有测试数据）。

## 5. 前端改造

- **设计器页**：`workflow/definition` 列表保留，新增全屏设计器路由 `workflow/designer/:id`。
  布局 = 左侧节点面板（拖入）+ 中间 LogicFlow 画布 + 右侧属性面板（选中节点/边时编辑
  审批人、会签或签、条件规则）。保存前跑与后端相同的图校验并在画布上标红问题节点。
- **自定义节点样式**：开始/结束（圆）、审批（矩形，显示审批人摘要）、会签（矩形加角标）、
  条件网关（菱形）、抄送（矩形虚线）。
- **表单设计**：定义编辑里加「表单字段」页签（字段列表增删排序即可，不做拖拽表单设计器）。
- **发起页**：按 `formFields` 渲染动态表单替代 textarea。
- **详情抽屉**：从 `el-steps` 升级为只读小地图（LogicFlow 只读模式渲染快照图，
  已过节点绿色/当前节点蓝色/驳回节点红色），审批记录表格沿用。

## 6. 实施阶段

| 阶段 | 内容 | 交付物 | 预估 |
|---|---|---|---|
| 一·图引擎 | JSON Schema 定稿；引擎改图遍历（start/approval/end，能力等价现状）；图校验；数据迁移 | 后端全部改造 + 单测（引擎纯逻辑适合作为项目首批单测） | 2~3 天 |
| 二·设计器 | LogicFlow 画布、自定义节点、属性面板、保存/回显/校验标红；详情只读小地图 | 设计器页替代节点列表编辑器 | 3~4 天 |
| 三·条件分支 | 表单字段定义 + 发起动态表单；条件网关 + 边规则求值 | 带分支的完整流程 | 2~3 天 |
| 四·增强 | 抄送节点与「抄送我的」列表；驳回策略（整体驳回/退回上一节点/退回发起人重提）；审批超时提醒（挂 Quartz） | 按需选做 | 每项 1~2 天 |

阶段一、二可并行度低（二依赖一的 Schema），建议顺序执行；阶段三依赖一二；阶段四独立可拆。

## 7. 风险与取舍

- **不做并行网关**（同时走多条分支再汇聚）：需要 token/汇聚计数模型，复杂度翻倍；
  多人同审已由会签覆盖，真实需求出现再评估。
- **不做子流程、脚本节点、消息节点**：超出后台管理框架定位。
- **LogicFlow 依赖体积**：core+extension 约 200KB gzip，设计器路由做异步 chunk，不影响首屏。
- **在途实例兼容**：靠快照机制天然隔离——引擎需同时认旧快照格式，或迁移时把在途实例快照一并转图（推荐后者，引擎只维护一种格式）。
